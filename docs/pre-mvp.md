# Pre-MVP PostPilot

## Зачем нужен Pre-MVP

Pre-MVP — это самый короткий путь проверить, что PostPilot решает реальную
проблему, **до** того как мы строим полный MVP из [MVP.md](./MVP.md).

Полный MVP описывает рабочий кабинет: редактор, AI, календарь, корзина
подтверждения, автопостинг и аналитика. Pre-MVP сознательно режет scope до
**одного замкнутого сценария**, который уже даёт пользу и отличает продукт от
«просто бота для постинга» из [idea.md](./idea.md).

**Гипотеза Pre-MVP:** администратор канала готов перейти с ручной публикации
(копировать текст → открыть Telegram → вставить → отправить) на единое место,
где он пишет пост и отправляет его в канал в один клик.

---

## Цель Pre-MVP

Дать одному администратору **одного** Telegram-канала минимальный рабочий
процесс:

> написал пост → сохранил черновик → опубликовал в канал через сервис → видит
> историю публикаций.

Если этот цикл работает и пользователь возвращается — можно наращивать MVP.

---

## Одна главная Job To Be Done

Из [troublesWeSolve.md](./troublesWeSolve.md):

**Когда я веду Telegram-канал регулярно, я хочу создавать и публиковать посты
из одного места, чтобы не тратить время на ручной copy-paste и не терять
черновики по заметкам и чатам.**

Pre-MVP закрывает только эту JTBD. Остальные (несколько каналов, аналитика,
регулярность через расписание, AI) — в MVP и дальше.

---

## Что входит в Pre-MVP

### 1. Минимальный вход

- регистрация / логин (один пользователь = один аккаунт);
- без команд, ролей и workspace;
- без биллинга и тарифов.

### 2. Подключение одного канала

- пользователь добавляет бота администратором в **один** канал;
- в кабинете указывает канал (username или ID);
- сервис проверяет, что бот — admin и может публиковать.

### 3. Простой редактор постов

- создание поста с текстом (plain text или базовый Markdown);
- ручное редактирование;
- сохранение черновиков;
- список черновиков и опубликованных постов.

**Без:** медиа-вложений, форматирования «как в Telegram», шаблонов, рубрик.

### 4. Публикация «сейчас»

- кнопка «Опубликовать» — единственная точка отправки;
- пост уходит в канал через Telegram Bot API;
- статус: `draft` → `published` / `failed`;
- сообщение об ошибке, если публикация не прошла.

**Корзина подтверждения из MVP здесь не нужна:** кнопка «Опубликовать» и есть
финальное подтверждение. Автопостинг по расписанию — уже MVP.

### 5. История публикаций

- список опубликованных постов: текст, дата, статус;
- ссылка на пост в Telegram (если API отдаёт `message_id`).

Это задел под аналитику MVP, но **без** просмотров, подписчиков и слотов по
времени.

---

## Что НЕ входит в Pre-MVP

| Функция | Куда переносим | Почему режем сейчас |
|---|---|---|
| AI-генерация и переписывание | MVP | Не блокирует core loop |
| Календарь и контент-план | MVP | Достаточно «опубликовать сейчас» |
| Отложенная публикация | MVP | Нужны очереди, worker, scheduler |
| Корзина подтверждения | MVP | При ручной публикации не нужна |
| Несколько каналов | MVP | Усложняет UI и модель данных |
| Медиа (фото, видео, файлы) | MVP | Текст закрывает 80% первых кейсов |
| Аналитика (просмотры, подписчики) | MVP | Нужен сбор метрик и фоновые jobs |
| Webhook для входящих updates | MVP | Для отправки достаточно `sendMessage` |
| Уведомления, audit log | MVP / Hardening | Не нужны для первых 5–10 пользователей |

---

## Чем Pre-MVP отличается от «ещё одного бота»

Из [idea.md](./idea.md) мы **не** строим «просто бот для постинга». Pre-MVP
всё равно остаётся **рабочим местом**, а не Telegram-командой:

- посты живут в сервисе как черновики, а не в заметках;
- есть история и статусы, а не «отправил и забыл»;
- интерфейс готов к расширению: календарь, AI, аналитика ложатся поверх тех
  же сущностей `Channel`, `Post`, `Publication`.

Pre-MVP — это **скелет pipeline**, а не финальный продукт.

---

## Для кого Pre-MVP

Из [clients.md](./clients.md), но уже:

- **1 пользователь → 1 канал**;
- личный автор или владелец тематического канала;
- ведёт канал регулярно и устал от ручного постинга.

Не целимся в агентства, сетки каналов и команды — это MVP и дальше.

---

## Упрощённый технический scope

Pre-MVP можно собрать **без** полной инфраструктуры из
[production-service-tech-plan.md](./production-service-tech-plan.md).

### Frontend

- React + Vite + TypeScript (уже есть `client/`);
- TanStack Query — списки постов, мутации publish/save;
- React Router или TanStack Router — 3–4 экрана;
- минимальный UI: shadcn/ui + Tailwind **или** простые стили на старте.

**Экраны:**

1. Login / Register
2. Подключение канала
3. Список постов (черновики + опубликованные)
4. Редактор поста

### Backend

- NestJS **или** один лёгкий API-слой (Fastify/Hono) — на выбор;
- PostgreSQL + Prisma;
- REST API;
- cookie-based auth;
- модуль `telegram`: только **исходящая** отправка (`sendMessage`).

**Не нужны на Pre-MVP:**

- Redis;
- BullMQ / worker;
- webhook endpoint;
- S3 (нет медиа);
- Sentry / OpenTelemetry (можно добавить перед первыми beta-пользователями).

### Модель данных (минимум)

```
User
  id, email, passwordHash, createdAt

Channel
  id, userId, telegramChatId, title, botConnectedAt

Post
  id, channelId, title?, body, status (draft | published | failed)
  publishedAt?, telegramMessageId?, errorMessage?, createdAt, updatedAt
```

### Telegram-интеграция

1. Пользователь добавляет бота admin в канал.
2. Указывает `@channel` или chat ID в настройках.
3. Backend проверяет права бота (`getChatMember`).
4. Publish → `sendMessage` → сохраняем `message_id` и статус.

Webhook и long polling **не нужны**, пока мы только отправляем, а не принимаем
updates.

---

## Порядок разработки Pre-MVP

Ориентир: **2–3 недели** одним разработчиком.

| Шаг | Что делаем | Результат |
|---|---|---|
| 1 | Auth + Prisma schema + docker-compose (Postgres) | Можно залогиниться |
| 2 | CRUD постов (черновики) | Редактор + список |
| 3 | Подключение канала + проверка бота | Канал привязан |
| 4 | Publish now через Telegram API | End-to-end работает |
| 5 | История + статусы ошибок | Пользователь видит результат |
| 6 | Полировка UI + деплой на staging | Можно отдать 3–5 beta-тестерам |

---

## Критерии успеха Pre-MVP

Pre-MVP считаем пройденным, если:

- [ ] 3–5 реальных админов каналов подключили свой канал;
- [ ] каждый опубликовал **минимум 5 постов** через сервис;
- [ ] **≥ 2** пользователя вернулись и опубликовали снова в течение 7 дней;
- [ ] есть хотя бы один качественный отзыв в духе «удобнее, чем вручную в
      Telegram».

Если пользователи не возвращаются — сначала докручиваем core loop, **не**
добавляем AI и календарь.

---

## Путь Pre-MVP → MVP

```text
Pre-MVP                          MVP
─────────────────────────────────────────────────────────
1 канал                    →     до 5 каналов
текстовые посты            →     медиа-вложения
publish now                →     расписание + worker
кнопка = подтверждение     →     корзина подтверждения
история публикаций         →     аналитика (просмотры, подписчики)
без AI                     →     AI: generate, rewrite, tone of voice
без календаря              →     календарь и контент-план
send-only Telegram         →     webhook + сбор метрик
минимальный auth           →     workspace, базовый RBAC
```

Каждый шаг MVP **наращивается** на те же сущности — не переписываем Pre-MVP с
нуля.

---

## Что сознательно откладываем даже после MVP

Из [MVP.md](./MVP.md) — без изменений:

- сложная командная работа;
- гибкие подписки;
- глубокая рекламная аналитика;
- геймификация и KPI-цели.

## Итог

| | Pre-MVP | MVP |
|---|---|---|
| **Срок** | 2–3 недели | 6–8 недель после Pre-MVP |
| **Цель** | Проверить core loop | Полноценный кабинет |
| **Каналов** | 1 | до 5 |
| **Публикация** | только «сейчас» | расписание + автопостинг |
| **AI** | нет | да |
| **Аналитика** | история | просмотры, подписчики, слоты |
| **Инфра** | Postgres + API | + Redis, BullMQ, webhook, observability |

**Pre-MVP = «написал → опубликовал → вижу историю» в одном канале.**

Если это работает — строим MVP. Если нет — меняем продукт, а не добавляем
фичи.
